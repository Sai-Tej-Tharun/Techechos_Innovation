The 7-Step Cloud Migration Playbook — That Actually Works

Moving to the cloud shouldn't be a gamble. Our battle-tested playbook — used across 20+ client migrations — walks you through every stage from first assessment to zero-downtime cutover.

Why Migrations Fail

We've taken over three failed cloud migrations from other agencies. The pattern is remarkably consistent: the technical execution was fine. The failure was in scope underestimation, insufficient rollback planning, and treating migration as a one-time project rather than a programme of work.

Step 1: Discovery & Assessment

Before you write a single line of Terraform, you need a complete inventory. Every service, every database, every external dependency, every API integration, every cron job, every file system dependency. We use a combination of automated discovery tools and manual interviews with every team that owns a service.

⚠️

The Hidden Dependency TrapThe services that break migrations are never the ones you know about. They're the legacy batch jobs, the direct database connections, the undocumented file shares. Budget 40% more discovery time than you think you need.

Classify Every Workload

We use a four-bucket classification: Retire, Retain, Rehost, and Refactor. Most teams skip straight to Refactor for everything, which is the most expensive mistake in cloud migration. Rehosting (lift-and-shift) 70% of workloads first, then optimising, is almost always the right strategy.

Step 2: Architecture Design

The target architecture must be designed before migration begins — not evolved during it. Key decisions that must be locked before you touch production:

  • VPC design, subnet strategy, and network segmentation
  • Identity and access management (IAM) structure and least-privilege policies
  • Landing zone architecture and account structure
  • Data residency and compliance boundaries
  • Shared services (logging, monitoring, DNS, secrets management)

Step 3: Foundation Building

Before any application migration, build the foundation using Infrastructure as Code. Everything in Terraform, no manual console actions. This ensures reproducibility, enables disaster recovery, and creates the audit trail your compliance team will eventually ask for.

# Example: Base VPC module we use for every migration
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"
  
  name = "${var.environment}-vpc"
  cidr = "10.0.0.0/16"
  
  azs             = data.aws_availability_zones.available.names
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]
  
  enable_nat_gateway   = true
  single_nat_gateway   = var.environment == "prod" ? false : true
  enable_dns_hostnames = true
}

Step 4: Migrate in Waves

Never migrate everything at once. Wave-based migration lets you learn from each wave before committing to the next. Wave 1 should be your lowest-risk, stateless services — ideally ones that are already containerised. Wave 2 introduces stateful services. Wave 3 tackles the legacy monoliths.

Step 5: Cutover Strategy

The cutover is where most migrations stumble. We use a Blue-Green DNS cutover approach with a 15-minute TTL change 48 hours before the actual cutover window. This allows DNS propagation to complete before traffic switches, and gives us a clean rollback path: flip DNS back if anything goes wrong.

✅

Zero-Downtime GuaranteeWe schedule cutovers on Tuesday or Wednesday mornings, never Fridays. We have a rollback runbook that can be executed in under 8 minutes. We've never missed our uptime SLA during a cutover in 20+ migrations.

Step 6: Optimise & Scale

Post-migration optimisation is where the cost savings appear. Reserved Instances for stable workloads, Spot for batch processing, right-sizing compute, S3 lifecycle policies, CloudFront caching — none of this belongs in the migration itself, but all of it should happen within 90 days of completing it.

Step 7: Ongoing Operations

Cloud is not infrastructure you deploy and forget. It requires ongoing care: cost monitoring and anomaly alerting, security posture reviews, dependency updates, capacity planning. We set up automated cost anomaly detection on day one — every client who skips this regrets it within three months.

Key Takeaways

What To Remember
  • Discovery always takes longer than you budget — plan for 40% overrun
  • Rehost first, optimise later — refactoring during migration is expensive
  • Everything in Infrastructure as Code before you touch production
  • Wave-based migration limits blast radius
  • Blue-Green DNS cutover with tested rollback runbooks
  • Post-migration optimisation is where the real cost savings live
  • Never schedule cutovers on Fridays
