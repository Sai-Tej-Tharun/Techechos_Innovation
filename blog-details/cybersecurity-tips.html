Enterprise clients have security teams that will ask hard questions. Most startups are embarrassingly unprepared. Here's the 40-point audit we run on every product before it goes near an enterprise deal.

Why This Matters More Than You Think

Enterprise security reviews have killed more SaaS deals than bad pricing. We've seen startups lose six-figure ARR contracts because they couldn't answer basic questions about data residency, encryption at rest, or their incident response process. These aren't advanced security topics — they're table stakes.

Authentication & Identity

Authentication is the most commonly misconfigured security control in SaaS applications. Our audit covers:

  • Password hashing: bcrypt, argon2, or scrypt with appropriate work factors — never MD5 or SHA1
  • Multi-factor authentication available (required for admin accounts, optional but encouraged for users)
  • JWT token lifetimes: access tokens ≤15 minutes, refresh tokens rotated on use
  • Session invalidation: all sessions invalidated on password change and account compromise
  • OAuth implementation: PKCE flow, no implicit flow, proper state parameter
  • Account enumeration prevention: identical error messages for invalid username and invalid password

Data Protection

Data protection failures are the ones that make the news. The controls we check:

  • Encryption at rest: AES-256 minimum for all databases and file storage
  • Encryption in transit: TLS 1.2+ everywhere, TLS 1.3 preferred, HSTS enabled
  • Secrets management: no secrets in environment variables in plain text, use a secrets manager (AWS Secrets Manager, Vault, etc.)
  • PII inventory: documented list of every field that contains personally identifiable information
  • Data retention and deletion: documented policy, automated enforcement
  • Backup encryption and access controls

API Security

APIs are the largest attack surface in most SaaS applications. The OWASP API Security Top 10 covers the critical risks. The controls we most commonly find missing:

Broken Object Level Authorization (BOLA)

The most common API vulnerability. Every API endpoint that returns data tied to a resource ID must verify that the authenticated user has permission to access that specific resource — not just that they're authenticated.

⚠️

BOLA ExampleGET /api/invoices/12345 — if a user can change 12345 to 12346 and see another customer's invoice, you have BOLA. This is the #1 API vulnerability and is almost always found in our audits.

Rate Limiting

Every public-facing endpoint must have rate limiting. Authentication endpoints must have aggressive rate limiting (5 requests per minute maximum). API endpoints should have business-logic-appropriate limits, not just technical ones.

Infrastructure Security

Infrastructure security controls we audit:

  • Principle of least privilege on all IAM roles and service accounts
  • No public S3 buckets containing non-public data
  • VPC configuration: application servers not reachable from the public internet directly
  • Security groups: deny all by default, allow by exception
  • CloudTrail or equivalent audit logging enabled
  • Automated vulnerability scanning in CI/CD pipeline
  • Dependency scanning: known CVEs flagged and blocked from deployment

Supply Chain & Dependencies

The SolarWinds and Log4Shell incidents changed enterprise security requirements permanently. Every enterprise security questionnaire now asks about supply chain controls. You need:

  • Software Bill of Materials (SBOM) generated for every release
  • Automated dependency scanning (Dependabot, Snyk, or equivalent)
  • Pinned dependency versions in production
  • Process for emergency security patches within 24 hours of critical CVEs

Compliance Foundations

SOC 2 Type II is the gold standard for SaaS security compliance in the US market. ISO 27001 for European markets. HIPAA for healthcare. GDPR for any EU personal data. The good news: the security controls required for all of these overlap significantly. Building to SOC 2 gets you 70% of the way to the others.

✅

Start NowSOC 2 Type II requires a minimum 6-month observation period. The best time to start your compliance programme was when you founded the company. The second best time is today.

Incident Response Readiness

Enterprise clients don't just ask "have you been breached?" — they ask "what would you do if you were?" Your incident response plan must be documented, tested, and accessible to the people who need it at 3am. At minimum: detection → containment → eradication → recovery → post-mortem.

Key Takeaways

What To Remember
  • Failed security reviews kill enterprise deals — prepare before you start selling upmarket
  • Authentication and BOLA are the most commonly misconfigured controls
  • Secrets management must use a dedicated secrets manager, not environment variables
  • Rate limiting is required on every public endpoint — not optional
  • Start your SOC 2 journey now — the 6-month observation period starts when you start
  • Incident response must be documented and tested before you need it
  • Supply chain controls are now required by every enterprise security questionnaire
